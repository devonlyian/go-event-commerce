# 스프링 하다가 Go 백엔드 해본 회고

스프링 백엔드를 오래 하다 보면 어느 순간 익숙함이 생긴다.  
프로젝트를 만들 때도 대략적인 그림이 머릿속에 있다. 컨트롤러를 만들고, 서비스와 리포지토리를 나누고, 설정을 붙이고, 실행해 보면 어느 정도는 예상한 대로 돌아간다. 프레임워크가 워낙 많은 걸 도와주기 때문에 생산성도 좋고, 큰 틀에서 길을 잃을 일도 적다.

그래서 이번에는 일부러 Go로 백엔드 프로젝트를 하나 끝까지 만들어봤다.  
문법만 훑는 정도로는 감이 안 올 것 같아서, 주문 생성 API 하나에서 끝내지 않고 이벤트 흐름까지 붙였다. 주문을 저장하고, 아웃박스에 이벤트를 기록하고, 카프카로 발행하고, 결제 워커가 그 이벤트를 소비하고, 다시 주문 상태를 갱신하는 흐름까지 만들었다. 여기에 도커 컴포즈, 쿠버네티스 매니페스트, 스웨거 문서, CI까지 붙였다.

결론부터 말하면, 생각보다 훨씬 재미있었고, 생각보다 훨씬 많이 배웠다.  
그리고 스프링을 오래 하다가 Go를 해보면 “문법이 다른 언어”를 배우는 느낌보다는 “백엔드가 실제로 어떻게 움직이는지 더 직접 보게 되는 경험”에 가깝다는 생각이 들었다.

## 왜 Go가 궁금했나

스프링은 정말 강력하다.  
하지만 오래 쓰다 보면 프레임워크가 너무 많은 걸 잘해줘서, 내가 실제로 어디까지 이해하고 있는지 흐려질 때가 있다. 예를 들면 의존성 주입, 애플리케이션 라이프사이클, 트랜잭션 경계, 메시지 소비 시점 같은 것들이다. 평소에는 잘 쓰고 있지만, 막상 “이게 정확히 어떻게 돌아가는가”를 설명하려고 하면 프레임워크 뒤에 숨어 있는 부분이 꽤 많다.

Go는 그 반대편에 있는 느낌이었다.  
프레임워크가 전체를 감싸주기보다, 필요한 것들을 직접 조합해서 써야 한다. 처음에는 불편할 수 있지만, 대신 애플리케이션이 시작되고 요청을 받고 종료되는 흐름이 코드에 그대로 드러난다. 나는 그 점이 궁금했다. 백엔드를 더 단순한 형태로 다시 보고 싶었다.

## 이번 프로젝트에서 만든 것

이번 프로젝트는 전자상거래 흐름을 아주 작게 잘라서 만든 연습용 시스템이다.

- 주문 API가 주문을 받는다
- 주문과 이벤트를 같은 트랜잭션으로 저장한다
- 아웃박스 이벤트를 카프카로 발행한다
- 결제 워커가 주문 이벤트를 소비한다
- 결제 결과 이벤트를 다시 발행한다
- 주문 서비스가 결제 결과를 구독해서 상태를 갱신한다
- 알림 서비스는 결제 결과를 받아 로그를 남긴다

처음에는 그냥 “Go로 API 하나 만들어보기” 정도로 시작했는데, 결국 비동기 이벤트 흐름과 서비스 간 연결까지 다 보게 됐다. 결과적으로는 단순 문법 연습보다는, 작은 이벤트 기반 시스템을 끝까지 만들어본 경험에 가까워졌다.

## 가장 먼저 느낀 차이

가장 크게 느낀 건, Go에서는 실행 흐름이 훨씬 노출되어 있다는 점이었다.

스프링에서는 애플리케이션이 돌아가는 기본 틀이 이미 잡혀 있다.  
어디에 무엇을 두면 되는지 익숙하고, 프레임워크가 객체를 만들고 연결하고 실행해준다. 덕분에 비즈니스에 집중하기 좋다.

반면 Go에서는 애플리케이션 시작점에서 직접 다 연결해야 했다.  
설정을 읽고, 로거를 만들고, 데이터베이스를 열고, 리포지토리를 만들고, 서비스를 만들고, 핸들러를 연결하고, HTTP 서버를 띄우고, 백그라운드 워커도 직접 시작해야 했다. 처음에는 꽤 번거롭게 느껴졌는데, 조금 지나고 나니 오히려 이게 장점처럼 느껴졌다.

무엇이 생성되고, 무엇이 어디에 주입되고, 어떤 순서로 실행되는지가 너무 명확했기 때문이다.  
스프링에서는 편해서 잘 안 보이던 부분이 Go에서는 다 보였다.

## “마법이 적다”는 말의 의미

Go를 두고 흔히 “마법이 적다”고 하는데, 이번 프로젝트를 하면서 그 말이 정확히 무슨 뜻인지 조금 알 것 같았다.

여기서 말하는 마법은 단순히 기능이 적다는 뜻이 아니었다.  
코드에 드러나지 않는 동작이 적다는 뜻에 가까웠다.

예를 들어 스프링에서는 어노테이션 하나로 해결되는 일들이 많다.  
의존성 주입도 그렇고, 컨트롤러 매핑도 그렇고, 설정 바인딩도 그렇고, 애플리케이션 라이프사이클도 그렇다. 익숙하면 정말 빠르고 편하다.

그런데 Go에서는 그런 것들이 대부분 코드로 드러난다.  
그래서 덜 편한 대신, 더 잘 보인다.  
그리고 이 “잘 보인다”는 감각이 초반엔 번거로움으로 오다가, 어느 순간부터는 이해도의 차이로 느껴졌다.

## 아웃박스와 메시지 처리를 더 현실적으로 보게 됐다

이번 프로젝트에서 가장 많이 배운 부분은 비동기 처리였다.

처음 구현할 때는 주문 저장 후 바로 카프카로 발행하면 끝일 것 같았다.  
그런데 조금만 생각해보면 데이터베이스에는 저장됐는데 메시지 발행이 실패할 수도 있다. 반대로 메시지는 나갔는데 상태 반영이 꼬일 수도 있다. 메시지를 소비한 뒤 커밋을 너무 일찍 해버리면 재처리가 안 되고, 너무 늦게 하면 중복 가능성이 생긴다.

스프링에서도 이런 문제는 물론 존재한다.  
하지만 실제로는 프레임워크와 라이브러리의 도움을 많이 받다 보니, 개념은 알아도 몸으로 부딪히는 느낌은 덜할 수 있다.

Go에서는 이걸 내가 직접 만져야 했다.

- 주문과 이벤트를 같이 저장해야 했다
- 발행 실패 시 아웃박스에 에러를 남겨야 했다
- 미발행 이벤트를 다시 보내는 릴레이 워커가 필요했다
- 메시지는 발행 성공 뒤에만 커밋하도록 바꿔야 했다
- 재처리해도 같은 결과가 나오도록 결제 로직을 결정적으로 만들어야 했다

이 과정에서 “데이터 저장”과 “이벤트 발행”은 같은 일이 아니라는 걸 더 또렷하게 이해하게 됐다.  
그리고 메시지 시스템은 단순히 라이브러리 호출 하나가 아니라, 실패와 재시도를 포함한 설계의 문제라는 점도 체감했다.

## 테스트를 보는 시선도 조금 바뀌었다

스프링을 할 때는 통합 테스트를 비교적 자연스럽게 많이 떠올리게 된다.  
컨텍스트를 띄우고, 필요한 컴포넌트를 주입받고, 실제 흐름을 보는 방식이 익숙하다.

Go에서는 그런 방향보다 더 잘게 쪼갠 단위 테스트가 먼저 손에 익었다.  
작은 인터페이스를 두고, 가짜 구현체를 만들고, 서비스 로직만 빠르게 검증하는 방식이 훨씬 가볍고 빠르다.

이번 프로젝트에서도 그 차이를 많이 느꼈다.  
서비스 로직은 fake 기반 테스트로 빠르게 확인하고, 진짜 연결은 도커로 올려서 마지막에 한 번 흐름을 검증했다. 이 방식은 실패 원인이 더 명확했고, 수정도 빨랐다.

스프링에서는 테스트가 “프레임워크 위에서 잘 도는가”를 많이 보게 된다면,  
Go에서는 “내 로직이 정확한가”를 더 직접적으로 보게 되는 느낌이었다.

## 문서와 CI를 붙이면서 든 생각

프로젝트가 어느 정도 돌아가고 나면 보통 기능을 더 붙이고 싶어진다.  
나도 처음에는 다른 도메인을 더 넣을까 생각했다. 그런데 막상 저장소를 하나의 결과물로 보니까, 부족한 건 기능보다 마감 품질이었다.

스웨거가 없으면 API를 남이 쓰기 어렵고,  
README가 약하면 처음 보는 사람이 프로젝트를 따라가기 어렵고,  
CI가 없으면 코드가 계속 살아있을 거라는 확신이 약하다.

그래서 마지막에는 기능 추가보다 문서화와 자동화에 시간을 썼다.  
그리고 이상하게도 이 단계에서 프로젝트가 비로소 “끝난 느낌”이 났다.

기능 구현만으로는 저장소가 완성되지 않는다.  
실행 방법이 정리되어 있고, 문서가 있고, 자동 검증이 돌아가야 비로소 남에게 보여줄 수 있는 상태가 된다. 이건 이번에 아주 분명하게 느낀 부분이다.

## 스프링 개발자 입장에서 낯설었던 부분

가장 낯설었던 건 어노테이션 중심 사고에서 벗어나는 일이었다.

스프링에서는 어노테이션이 구조를 설명한다.  
이 클래스가 어떤 역할인지, 어떻게 연결되는지, 어디에 매핑되는지 프레임워크 문법 안에서 읽힌다.

Go에서는 그런 설명을 코드 구조 자체가 해야 한다.  
패키지 이름, 생성자 함수, 명시적인 의존성, 함수 시그니처가 곧 구조 설명이다.

처음엔 표현력이 부족한 것처럼 느껴졌는데, 익숙해지고 나니 오히려 읽는 비용이 낮았다.  
무엇을 하겠다고 선언하는 방식보다, 무엇을 하는 코드가 바로 눈앞에 있는 방식에 가까웠다.

또 하나는 인터페이스를 바라보는 감각이었다.  
스프링에서는 인터페이스를 먼저 크게 잡는 일이 자연스러운데, Go에서는 필요한 쪽에서 작은 인터페이스를 두는 방식이 더 잘 맞았다. 이번 프로젝트에서도 테스트를 위해 작은 인터페이스를 도입하면서 이 차이를 꽤 실감했다.

## 이번 프로젝트를 마치고 남은 것

이 프로젝트를 하고 나서 가장 크게 남은 건 “Go로도 백엔드 서비스를 끝까지 만들어볼 수 있겠다”는 감각이었다.

이전에는 Go를 보면 문법이 낯설고, 스프링처럼 익숙한 구조가 없어서 멀게 느껴졌는데,  
지금은 적어도 이런 것들은 직접 해볼 수 있게 됐다.

- HTTP API 만들기
- 서비스와 리포지토리 나누기
- 명시적으로 의존성 연결하기
- 카프카 워커 만들기
- 아웃박스와 재시도 흐름 다루기
- 테스트 작성하기
- 스웨거, README, CI까지 붙여 마감하기

물론 아직 부족한 것도 많다.

- `context` 전파와 취소 처리
- 동시성 패턴
- 더 정교한 메시지 처리 설계
- 메트릭과 트레이싱
- 운영 환경에서의 장애 대응

하지만 이번 프로젝트는 적어도 “Go 문법을 조금 만져봤다” 수준은 확실히 넘었다.  
작은 규모라도, 백엔드 시스템을 직접 설계하고 끝까지 연결해본 경험으로 남았다.

## 그래서 연습용으로는 충분했나

내 판단은 그렇다. 충분했다.

오히려 여기서 도메인을 더 붙이는 건 학습 효율이 떨어질 가능성이 크다.  
이 프로젝트에서 이미 본 건 꽤 많다.

- 동기 API 처리
- 비동기 이벤트 흐름
- 데이터베이스와 메시지 브로커 연결
- 재시도와 상태 반영
- 문서화와 자동 검증

여기서 더 중요한 건 기능을 늘리는 것보다, 이번 경험을 정리하고 다음 문제로 넘어가는 것이다.  
다음에는 인증이 있는 서비스나, 캐시가 중요한 서비스나, 조회 성능이 더 중요한 서비스처럼 다른 종류의 문제를 가진 프로젝트를 해보는 편이 더 많이 남을 것 같다.

## 마무리

이번 Go 프로젝트를 하면서 가장 좋았던 건, 백엔드를 조금 더 맨몸으로 본 느낌이었다.

스프링은 여전히 강력하고 좋은 도구다.  
하지만 Go를 해보면서, 내가 평소 프레임워크의 도움을 많이 받고 있던 지점이 무엇인지 더 분명히 보이기 시작했다. 그건 스프링에 대한 부정이 아니라, 오히려 스프링을 더 잘 이해하게 된 경험에 가까웠다.

Go는 스프링보다 덜 친절할 수 있다.  
대신 그만큼 더 직접적이다.  
그리고 그 직접성이, 백엔드를 이해하는 데는 꽤 큰 도움이 됐다.

이번 프로젝트는 여기서 닫아도 충분하다.  
그리고 이제는 “Go를 공부해봤다”가 아니라, “Go로 작은 백엔드 시스템을 하나 끝까지 만들어봤다”라고 말해도 될 것 같다.
